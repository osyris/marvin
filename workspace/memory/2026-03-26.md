## Autonomous Dev Loop — концепция (обсуждение 26.03)

Файл: `/data/workspace/memory/dev-loop-concept.md`

### Суть
Постоянно работающий orchestrator читает задачи из центральной БД → запускает агентов → результат пишет обратно в БД → уведомляет KZ только при блокерах.

### Ключевые компоненты
- **БД:** Supabase (features table с полями status, plan, context_files, blocked_reason)
- **Orchestrator:** cron каждую минуту, считает параллельные задачи, спавнит воркеров
- **Complexity Assessor:** агент-роутер, решает simple vs complex pipeline
- **Simple pipeline:** implement → test → done
- **Complex pipeline:** plan validation → decomposition → parallel agents → integration test → done
- **Blocker handler:** пишет в БД + уведомляет KZ в Telegram
- **Watchdog:** проверяет зависшие in_progress задачи

### Стек
- Supabase + простой frontend (на нашей инфре /app/)
- OpenClaw cron как orchestrator
- ACP для запуска Claude Code агентов
- Я (Marvin) — принимаю задачи из разговора, пишу в БД, слежу за статусом

### Роль ACP в dev loop
KZ спросил подробно про ACP — проверил документацию:
- ACP запускает полноценный Claude Code (не headless) — slash-команды ✅, субагенты ✅, CLAUDE.md ✅
- **НО: ACP non-interactive** — нет TTY, нет ручного одобрения файлов/команд
- Права управляются через `permissionMode`: `approve-all` / `approve-reads` / `deny-all`
- Для autonomous dev loop это OK — агент работает с `approve-all`, human-in-the-loop только через блокеры в БД
- Для ручного просмотра каждого изменения — ACP не подходит, только интерактивный режим
